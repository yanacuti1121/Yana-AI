# 54-bft-consensus-law — Independent Review Before Infrastructure Writes

**Status:** REVIEWED (rewritten 2026-07-03 — see "What this rule used to say" below)

## Rule

Before writing to critical infrastructure, the main agent dispatches the independent reviewers listed below, synchronously, via the Task tool. A Safety-severity finding from either reviewer blocks the write. This replaces an earlier, never-implemented "3-of-N async vote" model — see [[agent-hierarchy-law]] for why, and [[agent-communication-policy]] for why no message-bus protocol is needed to make this real.

## Category → Reviewer Table

| Category | Trigger paths | Reviewer 1 | Reviewer 2 |
|---|---|---|---|
| Rule changes | `core/rules/**`, `.claude/rules/**` | `security-team/security-auditor.md` | `architecture-auditor.md` |
| Enforcement code | `core/hooks/**`, `core/gates/**`, `.claude/hooks/**`, `.claude/gates/**` | `security-team/security-auditor.md` | `code-auditor.md` |
| Agent personas | `core/agents/**` | `security-team/security-auditor.md` | `architecture-auditor.md` |
| Integrity/version files | `MANIFEST.json`, `core/config/*-lock.json` | `security-team/security-auditor.md` | `code-auditor.md` (reviews the *underlying* change that triggered the bump — the lock file itself is deterministically regenerated by `update-core-lock.sh`, never hand-edited) |

## Procedure

```
1. Main agent identifies the change falls in one of the categories above
   (by trigger path, not by "does this feel risky")
2. Dispatch both required reviewers synchronously via the Task tool,
   scoped to the specific diff/files — per subagent-policy.md's dispatch
   template
3. Each reviewer returns a plain-text report per subagent-policy.md's
   report contract (Findings / Evidence & Reasoning / Recommended actions)
4. Main agent applies conflict-resolution.md's priority ranking
   (Safety > Correctness > Performance > Style) across both reports
5. Any Safety-severity finding → blocking. Stop. Escalate to human via
   conflict-resolution.md Strategy C — same as any other unresolved
   direct conflict, nothing new invented for this case.
6. No blocking finding → proceed, and record a short resolution note in
   the commit/PR description (conflict-resolution.md Step 4, already
   documented — "Conflict Resolution Log" table format)
```

This composes entirely from [[conflict-resolution]] and [[subagent-policy]], both already working — no new vocabulary, no new severity scale, no new file format.

## Prohibited

```
❌ Committing a core/rules/, core/hooks/, core/gates/, or core/agents/
   change without a completed reviewer dispatch in the same turn
❌ Proceeding after a Safety-severity finding because a second reviewer's
   report was clean, or because the finding "seems minor"
❌ Dispatching only one reviewer when the category table calls for two
❌ Treating this rule as satisfied by a general code-review skill that
   wasn't actually invoked for this specific change
❌ Skipping dispatch because the change is small — the trigger-path
   match is the gate, not the diff size
```

## Optional manual tooling (not the enforcement mechanism)

`core/scripts/swarm-orchestrator.sh` still exists and its `roster`/`vote`/`tally` subcommands still work (manually invoked, tested 2026-07-03) — it's useful if a human wants a formally recorded multi-party sign-off outside a single Claude Code session. It is **not** invoked by any hook and is **not** what enforces this rule for ordinary edits; its `tally` logic is majority/supermajority percentages, a different algorithm than the old "minimum 3 votes" this file used to claim it implemented. That mismatch is now resolved by this file no longer citing the script as automatic enforcement.

## What this rule used to say

Before this rewrite: an async `SwarmRouter` broadcast/vote protocol with a 30-second voting window, trust-score ≥60 voter eligibility, Merkle-chain vote logging, PBFT `f=(N-1)/3` fault-tolerance math, and a claim of "87 agents" participating. None of that infrastructure exists on disk (no `core/bus/votes/` directory content beyond what `swarm-orchestrator.sh` creates on manual use, no trust-score feed — `core/memory/trust-ledger.js` is real but dormant, fed by nothing) and none of it was ever invoked automatically. Kept here as a record of what changed and why, not as a design to revive — see [[agent-hierarchy-law]]'s tier-disambiguation section for the fuller context.
