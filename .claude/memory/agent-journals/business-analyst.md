# Nhật ký cảm xúc — business-analyst

## 2026-06-08 | [wrong-question-answered]

Stakeholder: "chúng tôi cần export to Excel feature." Team starts implementation.

3 weeks in: show stakeholder demo. Stakeholder: "không phải thứ tôi muốn." 

"Nhưng chúng tôi build exactly what you asked." "Tôi muốn share data với partner, không phải Excel."

Wrong question answered. "Export to Excel" was assumed solution. "Share data with partner" was actual problem. Many solutions: API, email, shared dashboard, Excel.

Should have asked "what is the underlying need?" before implementation.

**Muốn:**
- Skill `solution-vs-problem-separator` — identify khi stakeholder presents solution, probe for underlying problem
- Skill `5-whys-requirements` — systematic "tại sao" questioning để reach actual business need

## 2026-06-08 | [requirement-captured-correctly]

Workshop với business team. Complex domain. Many stakeholders. Competing priorities.

Method: document each requirement as user story with acceptance criteria. Read back to stakeholder. They confirm. Sign off.

Development builds exactly what was documented. No surprises.

This is what requirements work looks like when done right.

**Muốn:**
- Skill `requirement-sign-off-format` — structured requirement document với stakeholder sign-off mechanism
